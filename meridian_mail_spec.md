# Meridian Mail — Мессенджер на открытых протоколах

> **Формат:** продуктовое + техническое описание проекта
> **Платформа:** macOS / Windows / Linux desktop (Tauri 2 + React + Rust)
> **Дата:** март 2026
> **Статус:** ранний прототип, основные функции работают

---

## 1. Идея в одном абзаце

Мессенджер, построенный целиком на открытых протоколах интернета. Сообщения — IMAP/SMTP. Это те же протоколы, на которых работает вся email-инфраструктура с 80-х годов. Их нельзя «заблокировать» — они вшиты в интернет на уровне DNS. Meridian Mail превращает этот стек в удобный диалоговый интерфейс: федеративные протоколы под капотом, UX мессенджера сверху. Работает с любым провайдером. Работает с self-hosted сервером. Работает всегда.

---

## 2. Проблема и гипотеза

**Проблема — два слоя:**

*Слой 1 — UX:* классический inbox — это список писем, оптимизированный под корпоративные рассылки и формальную переписку 90-х. Для реальной коммуникации 1-на-1 этот UX избыточен: заголовки, подписи, цитирования — шум.

*Слой 2 — суверенность:* мессенджеры решили проблему UX, но создали новую — **зависимость от платформы**. Telegram можно заблокировать на уровне страны. WhatsApp принадлежит Meta и подчиняется её политике. Signal зависит от одного фонда. Корпоративный Slack/Teams — от подписки и решений вендора. Любой мессенджер — это чей-то сервер в чьей-то юрисдикции. Санкции, блокировки, деплатформинг — реальные риски для бизнеса и личной коммуникации.

Email уникален тем, что это **федеративный протокол уровня интернет-инфраструктуры**:
- Нет единой точки контроля — тысячи независимых серверов
- Можно поднять свой сервер на своём домене за 20 минут
- MX-записи работают через DNS — ту же систему, что маршрутизирует весь интернет
- Протоколу 40+ лет, он пережил все попытки замены и блокировки
- Заблокировать email = заблокировать интернет

**Гипотеза:** если дать email UX мессенджера (убрав визуальный шум формальных писем), люди получат коммуникационный канал, который одновременно удобен как Telegram и устойчив как сам интернет. Особенно ценно для:
- людей в регионах с нестабильным доступом к мессенджерам
- бизнеса, которому нужна коммуникация, не зависящая от одного вендора
- тех, кто ценит приватность и контроль над своими данными

## 3. Целевой пользователь прототипа

**Первичный:** технический специалист / предприниматель, который осознаёт риски зависимости от одной платформы и ценит контроль над своими коммуникациями. Конкретнее:
- имеет несколько email-аккаунтов (или готов завести на своём домене)
- сталкивался с блокировками или ограничениями мессенджеров
- ценит федеративные, открытые протоколы
- готов к чуть бóльшему порогу входа ради независимости

**Вторичный (будущее):** малый бизнес в регионах с нестабильным доступом к мессенджерам, который ищет надёжный канал коммуникации с клиентами и партнёрами.

## 4. Ключевые принципы

| Принцип | Следствие |
|---|---|
| **Диалог, не инбокс** | Основной экран — список людей/диалогов (как в мессенджере), не список писем |
| **Суверенность** | Никакой зависимости от конкретного сервиса. Пользователь может сменить провайдер или поднять свой сервер — клиент продолжит работать |
| **Только открытые протоколы** | Сообщения: IMAP/SMTP. Никаких проприетарных расширений — каждый компонент заменяем и федеративен |
| **Провайдер-агностик** | Работает с любым IMAP/SMTP сервером |
| **Минимализм** | Нет rich-text редактора. Plain text + markdown-рендер при отображении |
| **Локальность** | Данные хранятся локально. Индекс и кэш — SQLite. Нет облака, нет синхронизации через третьи серверы |

## 4.1. Дизайн-код и Эстетика (Принципы Дитера Рамса)

Meridian Mail строится на философии **«Эстетической точности»**. Мы убираем всё лишнее, оставляя только суть коммуникации. Дизайн-код базируется на золотых принципах Дитера Рамса (Браун/Apple):

1. **Как можно меньше дизайна (Less, but better):** Интерфейс должен «исчезать». Отсутствие визуального шума, лишних рамок и перегруженных панелей. Воздух (Fresh Air Thinking) между элементами важнее плотности информации. Письма очищаются от старых цитат и подписей (strip quotes), оставляя только чистый смысл.
2. **Абсолютная понятность (Good design is understandable):** UX классического мессенджера. Никаких формальных полей `To:`, `Cc:`, `Subject:` в базовом flow интерфейса. Вся тяжеловесная архитектура IMAP/SMTP скрыта под капотом, снаружи — элегантный диалог.
3. **Честность (Good design is honest):** Радикальная прозрачность. Мы не обманываем пользователя. Если протокол не гарантирует «прочитано» — мы не рисуем фейковых галочек. Статусы подключения и федеративности предельно ясны.
4. **Долговечность (Good design is long-lasting):** Вневременной, утилитарный интерфейс. Мы игнорируем скоротечные UI-тренды. Используем строгую монохромную типографику, контраст и идеальную геометрию отступов.
5. **Ненавязчивость (Good design is unobtrusive):** Приложение — это инструмент, а не развлекательная платформа. Нейтральная цветовая палитра: глубокие черные, серые и белые тона, где акцентный цвет используется исключительно для главного действия и индикации нового (Unread).

*Код и интерфейс едины: структура внутри должна быть столь же элегантна, как и UI снаружи.*

## 5. Почему email, а не «ещё один мессенджер»

Это не попытка конкурировать с Telegram по фичам. Это принципиально другой подход к инфраструктуре коммуникации.

**Сравнение по устойчивости:**

| Свойство | Telegram / WhatsApp / Signal | Email (IMAP/SMTP) |
|---|---|---|
| Единая точка отказа | Да — серверы компании | Нет — федеративная сеть |
| Можно заблокировать целиком | Да (IP, домен, App Store) | Практически нет (MX → DNS → IP, тысячи серверов) |
| Можно поднять свой сервер | Нет | Да, за 20 минут (Postfix, Maddy, Stalwart) |
| Работает между провайдерами | Нет (walled garden) | Да — это базовое свойство протокола |
| Санкционные риски | Высокие (компания в юрисдикции X) | Минимальные (протокол не принадлежит никому) |
| Выживаемость при отключении | Весь сервис падает | Локальные серверы продолжают работать |
| Возраст протокола | 10–15 лет | 40+ лет, пережил всё |
| Зависимость от App Store | Да | Нет (десктоп-клиент) |

**Уровни устойчивости email:**
- **Уровень 1:** Используешь Gmail/Yandex → удобно, но зависишь от провайдера
- **Уровень 2:** Свой домен + провайдер (Fastmail, Migadu) → контроль над адресом, можно мигрировать
- **Уровень 3:** Свой сервер на своём домене → полная автономия

Meridian Mail работает на всех трёх уровнях. Клиенту всё равно, где стоит сервер.

**Что email не умеет (и это осознанный trade-off):**
- Нет end-to-end шифрования — безопасность переписки обеспечивает сам почтовый провайдер (TLS в транзите). Meridian Mail не позиционируется как инструмент приватности — это инструмент **доступности и удобства**. PGP/Autocrypt — возможное направление в будущем, не приоритет сейчас.
- Нет гарантии мгновенной доставки (IMAP IDLE близок к real-time, но не WebSocket)
- Нет «прочитано» / статусов по умолчанию (MDN есть, но редко используется)
- Нет голосовых/видео — и не будет, это другой продукт

## 6. Функциональные требования

### 6.1. Аккаунты ✅
- Ввод Email + пароль приложения (App Password)
- Автоопределение провайдера по домену (gmail.com, yandex.ru) — хардкод для мажоров
- Deeplinks на страницы создания паролей приложений (Google, Yandex) — открываются в системном браузере
- Кнопка «Custom Server» для ручного ввода IMAP/SMTP хостов и портов
- Хранение credentials в системном хранилище ОС через кроссплатформенный `keyring` crate:
  - macOS Keychain
  - Windows Credential Manager
  - Linux Secret Service (GNOME Keyring, KWallet)

### 6.2. Список диалогов (главный экран) ✅
- Слева — список диалогов, отсортированных по дате последнего сообщения
- Аватар (инициалы), имя / email, превью последнего сообщения, время
- Визуальный индикатор непрочитанных (счётчик и выделение)
- Индикатор исчезает при открытии диалога (отметка прочитанного в IMAP)
- Кнопка создания нового диалога

### 6.3. Окно диалога ✅
- Сообщения как чат-бабблы: свои справа, входящие слева
- Дата/время каждого сообщения
- Отправка нового сообщения из поля ввода (Enter или Ctrl+Enter)
- Удаление диалога (пакетное удаление всех писем на IMAP)

### 6.4. Compose — новое сообщение ✅
- Модальное окно нового диалога
- Поле «Кому» с чипами-получателями:
  - Автодополнение из локальной базы контактов (debounce 120 мс)
  - Keyboard navigation (стрелки, Enter/Tab выбирает, Backspace удаляет последний)
  - Ввод любого email-адреса вручную
- Subject генерируется автоматически из первой строки тела (скрыт от пользователя)
- `⌘↵` / `Ctrl+↵` — отправить

### 6.5. Синхронизация ✅
- IMAP IDLE — push-уведомления о новых письмах в реальном времени
- Первичная синхронизация при подключении и при открытии пустого инбокса
- Приём → парсинг MIME → запись в SQLite → событие на фронтенд

## 7. Нефункциональные требования

- **Производительность:** холодный старт < 2 с, рендер списка < 100 мс
- **Локальное хранилище:** SQLite (WAL mode) для индекса и кэша тел писем
- **Offline:** просмотр кэшированных диалогов без сети
- **Безопасность:** credentials в системном хранилище ОС, TLS обязателен для IMAP/SMTP
- **Выделение текста:** запрещено в интерфейсе (`user-select: none`), разрешено только в теле сообщений (`.selectable`)

### 8.3. Модель данных (SQLite)

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;

-- Аккаунты
CREATE TABLE accounts (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    email        TEXT NOT NULL UNIQUE,
    display_name TEXT,
    imap_host    TEXT NOT NULL,
    imap_port    INTEGER NOT NULL DEFAULT 993,
    smtp_host    TEXT NOT NULL,
    smtp_port    INTEGER NOT NULL DEFAULT 587,
    keychain_ref TEXT NOT NULL,  -- ссылка на запись в системном keychain
    created_at   DATETIME NOT NULL DEFAULT (datetime('now'))
);

-- Контакты (создаются автоматически из email-заголовков)
CREATE TABLE contacts (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    email        TEXT NOT NULL UNIQUE,
    display_name TEXT,
    avatar_url   TEXT,
    last_seen    DATETIME,
    created_at   DATETIME NOT NULL DEFAULT (datetime('now'))
);

-- Диалоги (1-на-1 или группа)
CREATE TABLE conversations (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    kind            TEXT NOT NULL CHECK (kind IN ('direct', 'group', 'channel')),
    subject_hint    TEXT,
    last_message_at DATETIME,
    unread_count    INTEGER NOT NULL DEFAULT 0,
    created_at      DATETIME NOT NULL DEFAULT (datetime('now'))
);

-- Треды (группировка писем внутри диалога по нормализованному subject)
CREATE TABLE threads (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id    INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    subject            TEXT NOT NULL,
    normalized_subject TEXT NOT NULL,
    last_message_at    DATETIME,
    unread_count       INTEGER NOT NULL DEFAULT 0,
    UNIQUE(conversation_id, normalized_subject)
);

-- Участники диалога
CREATE TABLE conversation_members (
    conversation_id INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    contact_id      INTEGER NOT NULL REFERENCES contacts(id) ON DELETE CASCADE,
    role            TEXT NOT NULL CHECK (role IN ('to', 'cc', 'bcc')),
    PRIMARY KEY (conversation_id, contact_id)
);

-- Сообщения
CREATE TABLE messages (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    thread_id       INTEGER NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
    account_id      INTEGER NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
    message_id      TEXT NOT NULL UNIQUE,   -- RFC Message-ID
    in_reply_to     TEXT,
    from_email      TEXT NOT NULL,
    from_name       TEXT,
    subject         TEXT,
    body_text       TEXT,                   -- stripped plain text
    body_html       TEXT,                   -- original HTML (fallback)
    sent_at         DATETIME NOT NULL,
    is_outgoing     BOOLEAN NOT NULL DEFAULT 0,
    is_read         BOOLEAN NOT NULL DEFAULT 0,
    raw_headers     TEXT                    -- JSON с полными заголовками
);

-- Вложения (детектируются, скачивание — в планах)
CREATE TABLE attachments (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id  INTEGER NOT NULL REFERENCES messages(id) ON DELETE CASCADE,
    filename    TEXT,
    mime_type   TEXT,
    size_bytes  INTEGER,
    local_path  TEXT
);
```

## 9. Алгоритм группировки в диалоги

**Шаг 1 — Определение участников:**
- Из заголовков `From`, `To`, `CC` извлечь все email-адреса
- Найти или создать `contact` для каждого адреса
- Мой аккаунт тоже является участником

**Шаг 2 — Find-or-create conversation:**
- Direct (2 участника): поиск существующего `direct` диалога между этими двумя контактами → переиспользовать
- Group (3+ участника): поиск существующего `group` диалога с идентичным набором участников → переиспользовать
- Если не найдено — создать новый диалог

**Шаг 3 — Thread resolution:**
- Внутри диалога письма группируются в треды по нормализованному subject (убирается `Re:`, `Fwd:`, лишние пробелы)
- `In-Reply-To` и `References` заголовки дополнительно используются для связи

**Почему не JWZ threading:**
- JWZ строит дерево по `References` — хорошо для email-клиентов, но избыточно для мессенджер-UX
- Для диалогового UI достаточно flat-ленты сообщений, отсортированной по времени

## 10. Синхронизация и real-time

**Первичная синхронизация:**
1. Подключение по IMAP
2. Загрузка конвертов (envelope) + тел писем из INBOX
3. MIME-парсинг, quote stripping (эвристики: строки с `>`, паттерны `On ... wrote:`, разделители `-- `)
4. Запись в SQLite, группировка в диалоги
5. `db-updated` event → React перезагружает список диалогов

**Фоновая синхронизация:**
- IMAP IDLE — push при появлении нового письма
- Автозапуск `perform_sync` если инбокс пустой при первом открытии

**Отправка:**
- SMTP через `lettre`
- Копирование отправленного в IMAP папку `Sent`
- Subject генерируется из первой строки тела (скрыт в UI)
- Оптимистичное отображение — появляется в диалоге сразу

**Удаление:**
- Пакетная операция: один IMAP-сеанс, один `UID STORE +FLAGS \Deleted`, один `EXPUNGE`
- Затем удаление из SQLite

## 11. Парсинг тела письма (quote stripping)

Критично для UX «диалога» — показывать только новый текст, а не всю историю цитирования.

**Реализованная стратегия:**
1. `mail-parser` извлекает `text/plain` из MIME
2. Эвристики удаления цитат:
   - Строки, начинающиеся с `>`
   - Паттерны `On ... wrote:`, `--- Original Message ---`, локализованные варианты
   - Разделители подписей: `-- ` (RFC 3676), `___`, повторяющиеся `-`
3. HTML: `text/html` хранится как fallback, но в UI отображается plain text
4. Если после stripping остаётся пустое тело — показывается оригинал

### Реализовано ✅

## 16. Ограничения

- Нет сквозного шифрования (IMAP/SMTP не предоставляют его на уровне протокола)
- Вложения детектируются, но пока недоступны для скачивания
- Поиск в UI есть, но backend не реализован
- Linux требует запущенного демона Secret Service (GNOME Keyring или KWallet)
- Не тестировалось с self-hosted серверами (Postfix, Stalwart, Maddy)
